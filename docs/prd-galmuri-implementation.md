## 1. Planning Summary

아키텍처 PRD (`prd-galmuri-engine-adapter.md`) 가 확정한 엔진/어댑터 구조를 구현 가능한 태스크 단위로 분해한다.

**핵심 변화**:
- `distill` = Core engine (본질환원 + 제1원칙 + 소크라테스 검증).
- 신규 어댑터 4개: `explain` / `pitch` / `doc` / `deck`.
- `deck` 내부 4 presets: `decision-sandwich-6` / `pitch-deck` / `concept-explain` / `story-arc`.
- `decide`, `shrink` 스킬 폐지 (하위호환은 트리거 라우팅으로 제공).
- 모든 신규 SKILL/reference 는 `.md` + `.ko.md` i18n 쌍.

**스케일**: 24 tasks / 7 phases / DAG 기반.

**성공 정의**: 구현자가 Phase 순서대로 태스크를 집어 작업 착수 가능. 각 태스크는 파일 경로, 변경 타입, acceptance criteria, 검증 명령을 자체 포함.

### Phase 1 — Engine (distill 재정의)

#### IMP-1. `scripts/parse-ratio.sh` (자연어 ratio 파싱 공용)

- **목적**: "절반/핵심만/한 줄 TL;DR" 같은 자연어를 숫자 ratio(0.05~0.5) 로 매핑. 모든 어댑터 공용.
- **산출**: `scripts/parse-ratio.sh` (실행 가능).
- **구현** (매칭 로직 명세):
  - 입력 정규화: `xargs` 로 leading/trailing whitespace 제거 + `tr '[:upper:]' '[:lower:]'` 로 소문자 변환.
  - 정규식 매칭 (GNU bash `[[ ... =~ ... ]]`), 순차 우선순위:
    1. `^(절반|half)$` → `0.5`
    2. `^(핵심만|핵심|essence|core[ ]?only)$` → `0.2` (띄어쓰기 선택)
    3. `^(한[ ]?줄|one[ ]?line|tl;?dr)$` → `0.05`
    4. `^0\.(0[5-9]|[1-4][0-9]?|5)$` → 숫자 그대로 (0.05 ≤ N ≤ 0.5)
    5. else → stderr `unknown ratio token: {input}`, exit 2.
  - 공백 변형 (`"절반 으로"`) 은 입력 정규화로 1 토큰화 불가 → 매칭 실패가 정상 동작. 조사 제거는 호출자(어댑터)의 선처리 책임.
- **Acceptance**:
  - `echo "절반" | bash scripts/parse-ratio.sh` → `0.5`
  - `bash scripts/parse-ratio.sh "0.2"` → `0.2`
  - `bash scripts/parse-ratio.sh "0.9"` → exit 2.
- **검증**: `tests/unit/parse-ratio.bats` (추후 IMP-22 에서 통합).

#### IMP-3. `skills/distill/references/decomposition.md(.ko.md)`

- **목적**: 제1원칙 분해법 (D/E/V/R) 가이드. decide 의 `SKILL.md` Step 2 로직을 reference 로 추출.
- **산출**: `skills/distill/references/decomposition.md` + `.ko.md`.
- **구현 (파일 본문 필수 포함 내용)**:

  1. **D/E/V/R 4 역할 정의** (각 1~2줄):
     - **D (Decision)**: 결정을 내리는 주체. "누가 이 명제를 받아들이거나 거부할 권한이 있는가?"
     - **E (Execution)**: 실행·집행하는 주체. "누가 이 명제를 실제 행동으로 옮기는가?"
     - **V (Validation)**: 검증·측정하는 주체. "누가 이 명제의 결과가 맞는지 확인하는가?"
     - **R (Recovery)**: 실패 시 복구·회복하는 주체. "누가 이 명제가 실패했을 때 원상복구하는가?"

  2. **각 역할 식별 질문 템플릿** (claim 1개당 4 질문 순차 적용):
     ```
     주어진 claim: "{claim}"
     Q_D: 이 claim 을 승인/거부할 수 있는 주체는 누구인가?
     Q_E: 이 claim 을 실행하는 주체는 누구인가?
     Q_V: 이 claim 의 결과를 검증하는 주체는 누구인가?
     Q_R: 이 claim 이 실패하면 복구 책임을 지는 주체는 누구인가?
     ```

  3. **Strict vs Weak 모드 표현 규약** (아키텍처 PRD §3.2 준수):
     - Strict: 4 역할의 subject 가 모두 distinct 해야 PASS.
     - Weak (`--weak-decomposition`): `role_*: "{subject} :: {perspective}"` 형태. subject 중복 허용, perspective 로 차이 표현. `EssenceUnit.decomposition.weak: true` 플래그 설정.

  4. **Strict 예시 (1개)**:
     ```json
     {
       "claim": "코드 리뷰는 4시간 내 완료한다",
       "decomposition": {
         "role_D": "팀 리드",
         "role_E": "리뷰 할당 개발자",
         "role_V": "CI 파이프라인",
         "role_R": "당직 엔지니어"
       }
     }
     ```

  5. **Weak 예시 (1개)**:
     ```json
     {
       "claim": "1인 스타트업은 MVP 를 2주 내 배포한다",
       "decomposition": {
         "weak": true,
         "role_D": "창업자 :: 기능 scope 결정 관점",
         "role_E": "창업자 :: 코딩/배포 관점",
         "role_V": "창업자 :: 사용자 반응 관측 관점",
         "role_R": "창업자 :: 장애 대응 관점"
       }
     }
     ```

- **Acceptance**:
  - 두 파일 모두 존재.
  - 4 역할 정의 섹션 존재 (`grep -c "^- \*\*[DEVR]" skills/distill/references/decomposition.md` ≥ 4).
  - Strict + Weak 예시 JSON 블록 각 1개 이상.
- **검증**: `test -f skills/distill/references/decomposition.md && test -f skills/distill/references/decomposition.ko.md && [ "$(grep -c '^- \*\*[DEVR]' skills/distill/references/decomposition.md)" -ge 4 ] && grep -q '"weak": true' skills/distill/references/decomposition.md`

## 7. Guardrails

### Project Guardrails (soft — 위반 시 경고)

- **태스크 순서 준수**: DAG 역행 금지. IMP-7~IMP-10 (어댑터) 은 IMP-6 (엔진 완성) 이후에만 시작. 급한 경우라도 stub 엔진으로 우회 금지.
- **i18n 동시 생성**: `*.md` 만 만들고 `*.ko.md` 를 뒤로 미루지 않음. 한 태스크 = 쌍 생성.
- **마이그레이션 diff 리포트 필수**: IMP-15, IMP-16 에서 자산 이관 시 반드시 diff 리포트 저장. "확인했음" 구두 OK 금지.
- **훅 매칭 대상 확장만 허용**: T19 에서 기존 훅 로직 변경 금지 (패턴만 확장). 로직 변경은 별도 PRD.
- **테스트 fixture 는 `tests/fixtures/` 에 집중**: 각 테스트 파일에 인라인 fixture 금지.
- **아키텍처 PRD §7 전부 상속**: 본 PRD 의 가드레일은 아키텍처 PRD 가드레일을 덮어쓰지 않음.
- **어댑터 prompt 에서 engine reference 복제 금지**: `skills/pitch/references/prompt.md` 등 어댑터 prompt 는 `skills/distill/references/socratic_probe.md` / `decomposition.md` / `essence-schema.json` 의 본문을 복제하지 말고 **파일 참조 링크만 사용**. A13/ARCH-T14 의 "단일 진실 원천" 불변식을 구현 중간 상태에서도 유지.

### Prohibitions (hard — 위반 시 즉시 중단)

- **바이너리 빌드 금지**: deck 어댑터 구현 시 PPTX/PDF/KEYNOTE 생성 로직 절대 도입 금지.
- **엔진 직접 UI 호출 금지**: distill SKILL 에 HITL 단계 추가 금지 (어댑터 책임).
- **essence_unit 스키마 무단 확장 금지**: 필드 추가는 본 PRD 수정 후만.
- **2개 이상 reference 동시 로드 금지** (Claude Code skill context budget).
- **decide / shrink 스킬 부활 금지**: 구현 중 편의상 임시 복원 금지. 이관 안 끝났으면 IMP-15/IMP-16 보완.
- **테스트 우회 머지 금지**: A1~A17 중 하나라도 실패 상태에서 메인 브랜치 머지 금지.

**완성 예시 — `skills/explain/SKILL.md`** (다른 어댑터 작성 시 이 레이아웃 준수):

```markdown
---
name: explain
description: >
  나 이해용 inline 설명 어댑터. distill 엔진을 호출하여 긴 텍스트의 본질을 뽑고 inline markdown 으로 렌더.
  audience 질의 없음, 저장 없음.
  Triggers: "설명해", "이해하게", "explain", "정리해서 보여줘", "readme 읽고"
version: 0.1.0
---

# galmuri:explain — 나 이해용 정리

## Prerequisites
- `scripts/preflight.sh` 통과 (jq, bats, bash).
- `skills/distill/` 엔진 설치됨.

## Step 1: Input capture
- 사용자 입력 (파일 경로 또는 stdin) 을 `.galmuri/tmp/source-{slug}.txt` 에 저장.
- audience 는 `me` 자동 고정 (질의 없음).

## Step 2: Engine invoke
> Claude Code skill 프레임워크에서 어댑터가 엔진을 호출하는 공식 경로는 **동일 세션 내 `distill` skill 을 Skill tool 로 호출** 하는 것. bash 직접 실행 파일 아님.
> 호출 시 전달해야 할 인자 (SKILL.md 프론트매터 / Step 지시로 수신):
>   - `--mode reduce`
>   - `--ratio 0.2`
>   - `--audience me`
>   - `--input` = Step 1 에서 저장한 source 경로
> 엔진 응답은 EngineOutput JSON (stdout) 으로 반환받고 Step 3 이 이어 렌더.

## Step 3: Render
- EngineOutput.units → inline markdown:
  - 각 unit 의 `essence` 를 bullet 로 나열.
  - 첫 unit 의 `claim` 을 상단 요약으로.

## Step 4: Output
- stdout 에 markdown 출력. **저장 단계 없음** (design intent).
- 세션 종료 시 `.galmuri/tmp/source-{slug}.txt` 자동 정리 (PostToolUse 훅).

## Output Schema
markdown body only. JSON 출력 없음.
```

다른 어댑터 (`pitch`, `doc`, `deck`) 는 위 구조에서:
- `pitch`: audience 질의 추가 + ratio 0.08 + hook-core-cta prompt + 저장 HITL.
- `doc`: audience 질의 + ratio 0.3 + 저장 기본 + PostToolUse 자산 기록.
- `deck`: preset 선택 HITL + JSON+markdown 2파일 저장 + 시각 지시 포함.
