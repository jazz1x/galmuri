## 1. Problem Definition

### 1.1 Current State (As-Is)

galmuri 는 현재 3개 독립 스킬로 구성됨: `decide` / `distill` / `shrink`.

**구조 역전**:
- 3가지 분해법(본질환원법 / 제1원칙 분해법 / 소크라테스식 검증) 은 galmuri 의 핵심 IP 지만, 현재 `decide` 스킬 한곳에 집중 구현되어 있음.
- `distill` / `shrink` 는 3 methods 를 부분적으로만 차용하며, 검증 게이트가 스킬별로 다르게 적용됨.
- **중복 실물 증거**: `skills/shrink/references/socratic_probe.md` 가 `decide` 의 probe 와 실질적으로 동일한 내용을 중복 보유.

**동기-구현 불일치**:
- galmuri 스킬화의 원래 동기는 "이해시키려면 메시지형이 베스트" — 타인 전달용 짧고 꽂히는 메시지(3~5줄) 산출.
- 그러나 `distill` 의 기본 출력 플로우는 `docs/galmuri-*.md` 저장 전제 → 문서형 산출에 가까워져 원 동기가 희석됨.
- "나 이해용" (설명 요청) 시나리오는 별도 스킬로 존재하지 않음.

**구조 제약**:
- `decide` 의 "6슬라이드 고정" 은 의사결정 덱에만 자연스러운 제약이지만 "Jobs-style 6-slide" 명명으로 일반화 가능한 표준처럼 표기됨.
- 실제 "Jobs 6슬라이드 프레임워크" 는 업계 표준이 아니며, 이 스킬의 자체 정의 결정 덱 구조. 네이밍 정직성 이슈.

### 1.2 Target State (To-Be)

- **Core engine 확립**: `distill` 이 3 methods 를 단독 보유하는 공용 엔진으로 재정의됨.
- **Adapter 분리**: `explain` / `pitch` / `doc` / `deck` 4개 어댑터가 엔진 출력을 받아 각각의 형태로 렌더링.
- **단일 검증 게이트**: 모든 어댑터는 동일한 소크라테스 검증 게이트 ("압축해도 본질 유지되는가") 를 통과한 essence unit 만 렌더링.
- **Deck 프리셋 가변화**: `decision-sandwich-6` (기존 decide 편입) / `concept-explain` / `pitch-deck` / `story-arc` 4개 프리셋. Open→Core→Close arc 고정, Core 길이 가변.
- **네이밍 정직성**: "Jobs" 이름은 디자인 토큰(타이포/컬러/레이아웃 미학) 에만 한정. 구조 명칭은 자체 용어.
- **하위호환**: 기존 `decide` 트리거 사용자는 `deck --preset decision-sandwich-6` 로 동일 산출물 획득 가능.

### 1.3 Business/Technical Impact

**지금 결정해야 하는 이유**:
- 원 동기(pitch 형태 퍼스트 클래스) 미구현 상태에서 `distill` / `shrink` 위에 추가 어댑터를 얹으면 중복이 기하급수로 증가.
- Socratic probe 의 이중 관리가 이미 발생 중 (`decide` + `shrink`). 어댑터가 늘면 N중 관리.
- 엔진 계약을 본 PRD 이후 단계로 미루면 각 어댑터의 기대 입력이 달라져 마이그레이션 비용 폭발.

**해결 안 할 시 비용**:
- 새 어댑터 1개 추가 ≈ 3 methods 재복제 비용 (현재 decide 수준).
- 트리거 자연어 공간 오염 ("압축" 이 distill/shrink/pitch 중 어디로 갈지 불명확).
- README / SKILL.md 상의 "Jobs-style" 표기가 사용자 기대를 오도.

## 2. Design Decision

### Alternative A: Full engine-adapter split (선택)

- **Approach**: `distill` = core engine (3 methods, essence unit 출력). `explain` / `pitch` / `doc` / `deck` = thin adapters. `deck` 내부 4 presets. `shrink` 는 독립 스킬 폐지, 엔진 공용 `--ratio` 옵션으로 흡수.
- **Pros**: 3 methods 단일 구현. 중복 제거. 새 어댑터 저비용. pitch 퍼스트 클래스. explain 신설. `decide` 는 `deck --preset decision-sandwich-6` 으로 하위호환.
- **Cons**: 마이그레이션 비용 (3 skills → 1 engine + 4 adapters). 트리거 네임스페이스 재배치. `--ratio` 파싱 공용 훅 설계 필요.
- **Suitable**: 재설계 원 동기와 1:1 매핑 가능. 본 PRD 시나리오.

### Alternative B: Minimal — shared references extraction only

- **Approach**: decide/distill/shrink 3 스킬 유지. Socratic probe + 공용 프롬프트만 `shared/references/` 로 추출.
- **Pros**: 마이그레이션 비용 거의 없음. 하위호환 완전.
- **Cons**: 원 동기(pitch) 미해결. explain 없음. "6슬라이드 고정" 제약 유지. 구조 역전 (3 methods 가 decide 집중) 그대로.

### Alternative C: Monolithic single skill

- **Approach**: `galmuri` 단일 스킬 + `--mode explain|pitch|doc|deck` + `--preset` 플래그.
- **Pros**: 멘탈모델 1개. 트리거 수렴.
- **Cons**: SKILL.md 비대 (모든 어댑터 분기 한 파일). 자연어 트리거 정밀도 저하. Claude Code skill 프레임워크의 "단일 목적" 규범 위반.

### Selection: Alternative A

- **Rationale**: 원 동기(pitch 퍼스트 클래스) + 3 methods IP화 + 중복 제거를 동시 달성하는 유일한 안. B 는 원 동기 미해결로 재설계 의미 상실. C 는 트리거 정밀도 희생이 galmuri 의 자연어 호출 UX 핵심 가치와 충돌.
- **Rejected alternatives and reasons**: B (원 동기 미해결), C (트리거 정밀도 희생)
- **Validity conditions**:
  - 어댑터 수가 8+ 로 늘어 엔진 공용화 이득이 SKILL.md 관리 비용보다 작아지면 재검토.
  - Claude Code skill 프레임워크가 subskill 구조를 공식 지원하면 subskill 로 이행.
  - `--ratio` 자연어 트리거("절반으로/압축/줄여") 가 4 어댑터 중 어디로 라우팅되는지에 대한 사용자 혼란이 실측 발생 시, 별도 `shrink` 어댑터 재분리 검토.
- **Referenced existing assets**: 없음 (첫 설계)

## 3. Technical Design

### 3.1 Architecture Overview

```
┌─────────────────────────────────────────────────────────────┐
│ User input (자연어 트리거)                                    │
└────────────────────┬────────────────────────────────────────┘
                     │
                     ▼
              ┌─────────────┐
              │  Adapter    │  (explain / pitch / doc / deck)
              │  (trigger   │   → audience / save policy / render
              │   owner)    │     규정. 엔진 호출 시 --mode + --ratio
              └──────┬──────┘                              전달
                     │
                     ▼
           ┌──────────────────────┐
           │  Core Engine:        │
           │  distill             │
           │                      │
           │  1. 본질환원법       │   reduce mode: 긴 텍스트 → essence
           │  2. 제1원칙 분해법   │   construct mode: 아이디어 → essence
           │  3. 소크라테스 검증  │   single gate: "압축 후 본질 유지?"
           │                      │
           │  Output:             │
           │    essence_units[]   │   (공용 계약, §3.2)
           └──────────┬───────────┘
                      │
                      ▼
              ┌───────────────┐
              │  Adapter      │   (엔진 출력을 어댑터 형태로 렌더)
              │  renderer     │
              └───────┬───────┘
                      │
                      ▼
                Output:
                  - explain: inline markdown (저장 없음)
                  - pitch:   3~5줄 메시지 (저장 선택제)
                  - doc:     docs/galmuri-{slug}.md (저장 기본)
                  - deck:    docs/galmuri-deck-{slug}.{json,md}
                             (JSON spec + 발표 스크립트 + 시각 지시)
```

**데이터 흐름 원칙**:
- 어댑터는 엔진 내부 상태에 의존하지 않음. essence_units 만 보고 렌더.
- 엔진은 어댑터의 렌더 형태를 모름. 동일 essence_units 가 4 어댑터 어디로든 갈 수 있어야 함.
- 검증 게이트 실패 시 엔진이 어댑터에 빈 출력 + 실패 사유를 반환. 어댑터는 HITL 로 처리.

### 3.2 Core Data Model

**essence_unit (공용 계약)**:

```typescript
interface EssenceUnit {
  id: string;               // "eu-{slug}-{n}"
  claim: string;            // "X 는 Y 다" 형태의 단일 명제
  essence: string;          // "누가 X하는가" 한 줄 (본질환원법 결과)
  decomposition: {          // 제1원칙 분해법 결과 (역할 분해)
    role_D: string;         // Decision (결정 주체)
    role_E: string;         // Execution (실행 주체)
    role_V: string;         // Validation (검증 주체)
    role_R: string;         // Recovery (회복 주체)
  };
  evidence: string[];       // 원문 근거 인용 (semantic grounding, §master.4.2)
  socratic_pass: {          // 소크라테스 검증 결과
    definition: boolean;    // Q1 answerable + explicit
    difference: boolean;    // Q2 answerable + explicit
    attribution: boolean;   // Q3 answerable + explicit
  };
  tags: string[];           // audience, domain (자유 문자열 키워드)
}

interface EngineOutput {
  units: EssenceUnit[];     // 검증 통과한 유닛만 포함
  mode: "reduce" | "construct";
  ratio?: number;           // reduce 모드 시 달성 압축 비율
  dropped: {                // 검증 실패로 버려진 항목
    claim: string;
    reason: "definition" | "difference" | "attribution" | "evidence";
  }[];
  source_ref: string;       // .galmuri/tmp/source-{slug}.txt
}
```

**Weak decomposition** (옵션):
- CLI 플래그 `--weak-decomposition` 활성 시 `role_D` / `role_E` / `role_V` / `role_R` 가 동일 주체라도 관점만 다르면 허용.
- 스키마 표현 규약:
  - 각 role 필드는 `"{subject} :: {perspective}"` 형태 문자열로 채움 (예: `role_D: "CTO :: 기술 방향 결정"`).
  - 4 필드 모두 required 유지 (optional 전환 금지). 관점 문자열로 차이를 표현.
  - 동일 subject 중복 허용을 표시하기 위해 `EssenceUnit.decomposition.weak: true` 플래그 추가 (strict 모드는 `false` 또는 필드 생략).
- Strict 모드 (기본) 는 subject 유일성 검증 통과 필수.

**deck preset 스키마**:

```typescript
interface DeckPreset {
  name: "pitch-deck" | "concept-explain" | "decision-sandwich-6" | "story-arc";
  arc: {
    open: OpenSlideSpec;     // 필수, dark 배경
    core: CoreSlideSpec[];   // 1~N, light 배경
    close: CloseSlideSpec;   // 필수, dark 배경
  };
  core_length: number;  // 1 (pitch) / 2-3 (concept) / 4 (decision) / essence_units.length (story)
  required_plugins: string[];        // e.g. decision-sandwich-6 → ["generalization-check"]
}

interface SlideSpec {
  slide_no: number;
  background: "dark" | "light";
  title: string;
  copy: string;
  visual_directive: string;   // 텍스트 지시만, 바이너리 빌드 없음
  parallel_rule?: string;     // same-position text length/POS match
}
```

### 3.3 Interface/API Design

**엔진 호출 시그니처**:

```
distill \
  --input <path|text> \
  --mode <reduce|construct> \
  [--ratio 0.05..0.5] \
  [--audience <string>] \
  [--weak-decomposition] \
  [--json]

출력: EngineOutput (JSON) 또는 markdown 렌더
```

**어댑터 호출 시그니처**:

```
explain <input>                  # mode=reduce, audience=me, save=never
pitch <input> [--audience X]     # mode=reduce, ratio≈0.1, save=prompt
doc <input> [--audience X] [--ratio R]  # mode=reduce, save=default
deck <input> --preset <name> [--mode decision|general]  # mode=construct (decision), reduce (otherwise)
```

**어댑터 → 엔진 전달 규약**:

| Adapter | mode       | audience default | ratio default | save default | verify depth |
|---------|-----------|-----------------|---------------|--------------|--------------|
| explain | reduce    | me              | 0.2           | never        | full         |
| pitch   | reduce    | (ask)           | 0.08          | prompt       | full         |
| doc     | reduce    | (ask)           | 0.3           | yes          | full         |
| deck    | construct | (ask)           | n/a           | yes          | full         |

**"verify depth = full" 의미**: 모든 어댑터가 소크라테스 검증 18문항 전부 통과 요구. 깊이 차등 없음 (유저 결정: "맥락이 압축되어도 유지되는가" 라는 본질 게이트는 출력 형태와 무관하게 동일).

**엔진 공통 훅**:
- `parse_ratio_natural_language`: "절반/핵심만/한 줄 TL;DR" → 0.5/0.2/0.05 매핑. 모든 어댑터 공용.
- `audience_prompt`: audience 미명시 시 자연어 질의. `explain` 은 `me` 로 자동 설정하여 스킵.
- `essence_unit_validator`: EngineOutput 스키마 검증. 어댑터 진입 전 강제.

### 4.2 Function/Component Signatures

**Engine API (distill SKILL.md Step 구조)**:

```
Step 1: Input capture
  - --input 경로 또는 stdin 텍스트 → .galmuri/tmp/source-{slug}.txt

Step 2: Mode selection
  - --mode reduce: 기존 텍스트 압축 (입력은 긴 텍스트 전제)
  - --mode construct: 아이디어/주제로부터 본질 구성 (입력은 짧은 프롬프트)
  - 미지정 시 자동 추정 규칙:
      * source_tokens < 80  → construct 제안
      * source_tokens >= 80 → reduce 제안
      * (토큰 카운트는 `scripts/count-tokens.sh` 기준)
    추정 결과를 HITL 로 제시하여 confirm 후 진입 (silent default 금지).

Step 3: Method application (순서)
  3a. 본질환원법 → 후보 essence 문장들 (각 명제별)
  3b. 제1원칙 분해법 → 각 essence 를 D/E/V/R 로 분해 (또는 weak)
  3c. 소크라테스 검증 → Definition/Difference/Attribution 18문항
      - 실패 항목은 dropped[] 로 기록 후 제외

Step 4: Ratio fitting (reduce 모드만)
  - --ratio 주어지면 target_tokens 계산
  - 통과한 units 합이 target 초과 시 낮은 evidence 점수 순으로 드롭
  - |actual - target| > 0.05 → 최대 2회 재시도 → HITL

Step 5: Output
  - EngineOutput JSON 또는 markdown 요약
  - 어댑터 호출된 경우 파이프로 전달
```

**Adapter API (예: pitch)**:

```
Step 1: Audience
  - --audience 없으면 자연어 질의: "누구한테 꽂아야 하나요?"

Step 2: Engine invoke
  - distill --mode reduce --ratio 0.08 --audience {X}

Step 3: Render
  - EngineOutput.units → hook-core-cta 3비트 구조로 3~5줄 렌더
  - references/prompt.md 지시 적용

Step 4: HITL
  - inline 출력. "저장할까요? (y/n/edit-slug)" 질문.
  - y → docs/galmuri-pitch-{slug}.md

Step 5 (optional): Asset record
  - 저장 시 scripts/record-asset.sh --type pitch 호출
```

**Deck preset 라우팅**:

```
deck --preset <name> <input>
  ├─ decision-sandwich-6 → mode=construct, 6 slides, Generalization plugin 활성
  ├─ pitch-deck          → mode=reduce, 3 slides, ratio=0.1
  ├─ concept-explain     → mode=reduce, 4-5 slides, ratio=0.25
  └─ story-arc           → mode=reduce, N=essence_units.length slides, ratio user-defined
```

### 4.3 State Management Approach

- **Source capture**: `.galmuri/tmp/source-{slug}.txt` (기존 방식 유지). PreToolUse 훅이 생성, PostToolUse 훅이 정리.
- **Engine output**: 휘발성. JSON 파이프로 어댑터 전달. 저장 안 함.
- **Adapter output**: 어댑터별 저장 정책 따름 (§3.3 표 참조).
- **Asset record**: `.harnish/assets/*.jsonl` (harnish ecosystem 공존). 저장 성공 시에만.
- **No global state**: 스킬 간 공유 전역 상태 없음. 모든 통신은 파일 / stdout 파이프.

## 5. Edge Cases & Error Handling

| Scenario | Expected Behavior | Handling Strategy |
|----------|------------------|-------------------|
| 입력이 reduce 하기에 너무 짧음 (source_tokens < 80, §4.2 Step 2 기준) | 엔진이 mode 자동 재추정 construct 제안 | HITL: "입력이 {N} 토큰으로 짧습니다 (임계 80). construct 모드로 아이디어 구성할까요? (y/n)" |
| 소크라테스 검증 전부 실패 (units = []) | 빈 EngineOutput + 실패 사유 | 어댑터는 렌더 대신 HITL: "본질 추출 실패: {사유}. 입력을 더 구체화하거나 수동 검토 필요." |
| ratio 수렴 실패 (2회 재시도 후 편차 > 0.05) | HITL 제공 | `[a]ccept / [r]e-target / [c]ancel` |
| `decide` 트리거로 진입 (하위호환) | `deck --preset decision-sandwich-6` 로 자동 라우팅 | 트리거 매칭 레이어에서 alias. 사용자에게 "deprecated, use deck" 경고 1회. |
| `shrink` 트리거로 진입 (하위호환) | 입력 문맥으로 어댑터 자동 선택 | "절반으로" + 긴 텍스트 → doc. "한 줄 TL;DR" → pitch. 모호 시 HITL. |

| 동일 slug 중복 저장 시도 | 충돌 경고 | `docs/galmuri-{slug}-v{n}.md` 로 증분 저장 제안. |

## 7. Guardrails

### Project Guardrails (soft — 위반 시 경고)

- **엔진 외부 의존 최소화**: `distill` 엔진은 LLM 호출 + bash 스크립트 + 표준 유닉스 툴만 사용. 새 외부 라이브러리 도입 금지 (합의 없이).
- **어댑터는 엔진 내부를 모름**: 어댑터는 EngineOutput 스키마만 보고 렌더해야 함. `distill/scripts/` 내부 함수 직접 호출 금지.
- **검증 게이트 단일화**: 모든 어댑터는 `distill` 의 소크라테스 게이트만 사용. 어댑터에 별도 검증 로직 추가 금지.
- **Ratio 자연어 파싱 공용**: 어댑터별로 ratio 파싱 코드 복제 금지. `scripts/parse-ratio.sh` 만 사용.
- **Preset 플러그인 명시**: `decision-sandwich-6` 의 Generalization 같은 preset-specific 로직은 `required_plugins` 메타데이터로 선언. 하드코드 금지.
- **저장 경로 규약**: 어댑터 저장은 `docs/galmuri-{adapter}-{slug}.{ext}` 형식. 예외 금지.

### Prohibitions (hard — 위반 시 즉시 중단)

- **바이너리 빌드 금지**: deck 어댑터는 PPTX/PDF/KEYNOTE 바이너리 포맷을 생성하지 않음. JSON spec + markdown + 시각 지시 텍스트만.
- **엔진 직접 UI 호출 금지**: 엔진은 HITL 을 직접 수행하지 않음. 모든 사용자 상호작용은 어댑터 책임.
- **decide / shrink 스킬 부활 금지**: 이 PRD 의 핵심 결정. 재도입은 별도 PRD 필요.
- **"Jobs 6-slide" 명명 재도입 금지**: 구조 명칭과 디자인 토큰 명칭 분리 규칙은 철회 불가 (네이밍 정직성).
- **essence_unit 스키마 무단 확장 금지**: 필드 추가는 마이너 버전 bump + 전 어댑터 영향 검토 필수.
- **2개 이상 reference 동시 로드 금지** (Claude Code skill context budget 규범 상속).
